# Why the previous versions failed

This section is blunt on purpose.

## 1. The UX started from internal system nouns

The old UI forced the user to think in terms of campaigns, prospects, and outreach objects before anything useful happened.

That is not how founders or sales operators think when they are trying to make something happen.

They start from a messy human thought like:

- "I want to do business with Hacker News."
- "I want to understand whether Andreessen is the right investment target."
- "I think this company has a product wedge here, but I need the angle."

The system should meet the user there.

## 2. The product behaved like a copy generator with extra steps

The old design showed strategies, signals, and drafts, but the chain from:

- target resolution
- context research
- surface mapping
- outreach writing

was not clear enough.

That made the diligence layer feel decorative rather than essential.

## 3. The output was incomplete

Subject lines and titles were not treated as first-class deliverables.

That is a major miss for cold outreach.

A tool that claims to help with outreach has to own:

- title / framing
- subject line
- body draft
- follow-up
- channel variation

## 4. The input asked for a machine-friendly prompt

The user was effectively being asked to do the agent's job.

The correct input is not:

- "Sell Algolia search to Paul Graham for Hacker News..."

The correct input is closer to:

- "pg / Hacker News / I want a business angle here"
- "Andreessen / a16z / I want investment-fit diligence first"

The system should turn rough human intent into structured research work.

## 5. Too many controls were shown too early

The old version over-exposed strategy cards, signal toggles, and pack UI before it had earned trust.

The user does not need a cockpit. The user needs one clear answer:

- who is the target really
- what surface matters
- what wedge is defensible
- what should I send

## 6. The backend was overbuilt relative to the request

The ask here is to get the service definition, UX, and agent plan right.

The backend and worker should be defined as contracts and operating plans, not prematurely implemented.

## Reset rule

The product now follows this rule:

**front-end must make the workflow obvious before any backend exists**
